Raids System
Party-based boss fights over 2 / 4 / 7 days — permanent gear sacrifice required
New
How Raids Work
Form a party of up to 4 players. The party leader selects a tier and everyone sacrifices gear as tribute — items are permanently removed from inventory. Each day, every party member gets one combat session against the raid boss using the real arena combat engine. Damage accumulates in a shared boss HP pool across all party members until the boss dies or time runs out.

⚙ Technical Details
Combat uses the existing combat.ts engine (computeBattleStateAtTimeWithFood + simulateBattleWithFood) — no new combat code written. Each session runs a 120-second fight in RaidFightModal with real-time HP bars. Boss contributions stored in Supabase raid_participants.daily_attacks JSONB. Raid Medals stored in profiles.raid_medals.

Guild System (v3.7–3.8)
Full guild management with invites, tax, member roles, and treasury
New in v3.7
Phase 1 — Core Guild Features
Create/join/leave guilds. Guild treasury (deposit gold). Weekly collective goal progress tracked via Supabase RPC. Activity log showing member events. Full member list with roles (owner/officer/member) and contribution stats.
Phase 2 — Member Management
Kick (owner + officer) with inline ✓ ✕ confirmation. Promote to officer / demote to member (owner only). "Show more/less" toggle when members exceed 12. Optimistic local state updates on all actions.
Phase 3 — Invites & Tax
Guild Invites: Officers+ can invite friends via profile page. Pending invites shown in GuildTab — accept auto-joins the guild. Guild Tax: owner sets 0–15% via slider. Auto-deducted from arena gold on mob kills and routed to guild treasury.

Database Migrations
Two new Supabase tables + column addition
📋 Migration 1 — add_raids_table
Creates public.raids table:
id UUID PK · tier (ancient/mythic/eternal) · leader_id → profiles · status (active/won/failed) · boss_hp_remaining BIGINT · boss_hp_max BIGINT · started_at / ends_at TIMESTAMPTZ · tribute_items JSONB

RLS: SELECT — leader or participant. INSERT — leader only. UPDATE — leader or participant.
📋 Migration 2 — add_raid_participants_table
Creates public.raid_participants table:
raid_id + user_id composite PK · username · joined_at · tribute_paid BOOL · daily_attacks JSONB

Adds raid_medals INTEGER DEFAULT 0 to profiles.

RLS: SELECT — any participant of the raid. INSERT — own user_id only. UPDATE — own user_id only.
